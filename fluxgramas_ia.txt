01) Desenhe para mim um fluxograma que faça:

1 - a relação de uma aplicação Angular SPA com um microsserviço Quarkus;
2 - deve-se autentificar via LDAP usando keycloak;
3 - as requisições entre a aplicação web e o microsserviço devem ser feitas usando REST API + JWT + TOKEN (usar o provider token), podem ser feitas também usando websocket e mensageria;
4 - um fluxograma deve ser o padrão de sequência;

Claro — para o seu caso, o mais adequado é um **diagrama de sequência** mostrando a interação entre:

- **Usuário**
- **Angular SPA**
- **Keycloak**
- **LDAP**
- **Microsserviço Quarkus**
- opcionalmente **WebSocket**
- opcionalmente **Broker de Mensageria**

Segue um fluxograma no **padrão de sequência** em **Mermaid**, que você pode colar em ferramentas compatíveis:

```
sequenceDiagram
    actor U as Usuário
    participant A as Angular SPA
    participant K as Keycloak
    participant L as LDAP
    participant Q as Microsserviço Quarkus
    participant W as WebSocket Channel
    participant M as Broker/Mensageria

    U->>A: Acessa a aplicação
    A->>K: Redireciona para autenticação
    K->>L: Valida credenciais no LDAP
    L-->>K: Credenciais válidas
    K-->>A: Retorna Access Token JWT + Refresh Token
    A-->>U: Usuário autenticado

    U->>A: Executa ação na SPA
    A->>A: Obtém token do provider de token
    A->>Q: Requisição REST API com Authorization: Bearer JWT
    Q->>K: Valida token / obtém chaves públicas
    K-->>Q: Token válido
    Q-->>A: Resposta da API
    A-->>U: Exibe resultado

    opt Comunicação em WebSocket
        A->>W: Conecta com token JWT
        W->>K: Valida token
        K-->>W: Token válido
        W-->>A: Canal autenticado
        Q->>W: Publica evento em tempo real
        W-->>A: Envia atualização em tempo real
    end

    opt Comunicação assíncrona por mensageria
        Q->>M: Publica mensagem/evento
        M-->>Q: Confirma recebimento
    end
```


### Leitura do fluxo
1. O usuário entra na **Angular SPA**.  
2. A SPA redireciona para o **Keycloak**.  
3. O **Keycloak** autentica o usuário contra o **LDAP**.  
4. Após sucesso, o Keycloak devolve um **JWT Access Token** e possivelmente um **Refresh Token**.  
5. A SPA usa esse token nas chamadas para o **microsserviço Quarkus** via **REST API**.  
6. O Quarkus valida o token recebido.  
7. Opcionalmente, o sistema também pode se comunicar por:
   - **WebSocket**, para tempo real;
   - **Mensageria**, para processamento assíncrono.

4)

Faça um fluxograma comparativo com dois lados opostos laterais com suas medidas físicas proporcais onde:

Lado A:

Arquitetura Antiga
* Frontend
    - Backbone*
    - jQuery*
    - front vinculado ao código Java (monolítico)

* Backend
    - Implementações duplicadas de alguns processos defenidos via interface
    - Implementações restritas à única arquitetura monolítica
    - JVM mais lenta e com o código 100% interpretado
    - GC antigo, lento e com maior alocação de recursos
    - Código mais verbose
    - Domínios restritos aos DTOs e Enums
    - Threads comuns de SO com blocks mais constantes e que usam mais recursos
    - Arquitetura batch presa ao Control-M e Spring Batch

Lado B:

Arquitetura Microsserviços
* Frontend
    - Angular
    - TypeScript e JS moderno padrão ES2025
    - front desacoplado e independente do código backend (microsserviços)

* Backend
    - Implementações únicas e não duplicadas. Quer consumir um recurso? Use o seu microsserviço.
    - Implementaçòes escaláveis a diferentes serviços
    - JVM moderna e com a possibilidade de rodar em código nativo e JIT.
    - GC antigo, lento e com maior alocação de recursos
    - GC modelo novo ZGC mais de 10x mais rápido que GC antigas
    - Código mais limpo, menos verbose
    - Domínios em DTOs, Enums e Records (mais direto e com auto equals, hashCode e toString)
    - Virtual Threads permitem carregar mais processos e com menos recursos
    - Arquitetura batch usando Quarkus + Scheduler (Temporal.io), Quarkus Batch (JBeret)
