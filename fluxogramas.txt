01) Modelo de fluxo de autenticação do usuário na aplicação

sequenceDiagram
    actor U as Usuário
    participant A as Angular SPA
    participant K as Keycloak
    participant L as LDAP
    participant Q as Microsserviço Quarkus
    participant W as WebSocket Channel
    participant M as Broker/Mensageria

    U->>A: Acessa a aplicação
    A->>K: Redireciona para autenticação
    K->>L: Valida credenciais no LDAP
    L-->>K: Credenciais válidas
    K-->>A: Retorna Access Token JWT + Refresh Token
    A-->>U: Usuário autenticado

    U->>A: Executa ação na SPA
    A->>A: Obtém token do provider de token
    A->>Q: Requisição REST API com Authorization: Bearer JWT
    Q->>K: Valida token / obtém chaves públicas
    K-->>Q: Token válido
    Q-->>A: Resposta da API
    A-->>U: Exibe resultado

    opt Comunicação em WebSocket
        A->>W: Conecta com token JWT
        W->>K: Valida token
        K-->>W: Token válido
        W-->>A: Canal autenticado
        Q->>W: Publica evento em tempo real
        W-->>A: Envia atualização em tempo real
    end

    opt Comunicação assíncrona por mensageria
        Q->>M: Publica mensagem/evento
        M-->>Q: Confirma recebimento
    end

04) Comparativo de arquiteturas

flowchart TD

    T[Comparativo de Arquiteturas]

    T --> H1
    H1 --> A0
    H1 --> B0

    H1[Visão Geral]
    A0[Arquitetura Antiga]
    B0[Arquitetura Microsserviços]

    A0 --> A1
    B0 --> B1

    A1[Frontend<br/>Backbone + jQuery<br/>frontend vinculado ao código Java monolítico]
    B1[Frontend<br/>Angular<br/>TypeScript + JS moderno ES2025<br/>frontend desacoplado e independente do backend]

    A1 --> A2
    B1 --> B2

    A2[Implementação de backend<br/>implementações duplicadas de processos definidos por interface<br/>restritas à única arquitetura monolítica]
    B2[Implementação de backend<br/>implementações únicas e não duplicadas<br/>consumo por microsserviço responsável<br/>escalável para diferentes serviços]

    A2 --> A3
    B2 --> B3

    A3[JVM<br/>mais lenta e com menos recursos modernos]
    B3[JVM<br/>moderna com suporte a JIT e possibilidade de código nativo]

    A3 --> A4
    B3 --> B4

    A4[Garbage Collector<br/>GC antigo, mais lento e com maior alocação de recursos]
    B4[Garbage Collector<br/>ZGC moderno, de baixíssima latência e mais de 10x mais rápido que coletores antigos]

    A4 --> A5
    B4 --> B5

    A5[Código<br/>mais verbose]
    B5[Código<br/>mais limpo e menos verbose]

    A5 --> A6
    B5 --> B6

    A6[Modelagem de domínio<br/>domínios restritos a DTOs e Enums]
    B6[Modelagem de domínio<br/>DTOs, Enums e Records<br/>modelagem mais direta com equals, hashCode e toString automáticos]

    A6 --> A7
    B6 --> B7

    A7[Concorrência<br/>threads comuns do SO<br/>mais bloqueios e maior uso de recursos]
    B7[Concorrência<br/>Virtual Threads<br/>mais processos com menos recursos]

    A7 --> A8
    B7 --> B8

    A8[Batch<br/>arquitetura presa ao Control-M e Spring Batch]
    B8[Batch<br/>Quarkus + Scheduler<br/>Temporal.io / Quarkus Batch / JBeret]
